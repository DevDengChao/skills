# Upgrade Dependencies

使用仓库实际采用的包管理器、构建工具或封装脚本检查过时依赖，先规划升级与验证，再按最小可验证批次执行。目标是稳定推进，而不是一次性升级全部依赖后集中排错。

## 建立上下文

- 读取当前目录及相关上游目录中的 `AGENTS.md`。
- 识别仓库结构、模块边界和锁文件策略；优先查看实际清单文件、锁文件、wrapper 脚本和工作区配置。
- 默认识别这些入口：
  - Node：`package.json`、`pnpm-lock.yaml`、`package-lock.json`、`yarn.lock`、`bun.lockb`、`pnpm-workspace.yaml`
  - Python：`pyproject.toml`、`requirements*.txt`、`poetry.lock`、`uv.lock`
  - Rust：`Cargo.toml`、`Cargo.lock`
  - Go：`go.mod`、`go.sum`
  - PHP：`composer.json`、`composer.lock`
  - Java：`pom.xml`、`build.gradle`、`build.gradle.kts`、`settings.gradle`、`settings.gradle.kts`
- 读取仓库已有的验证入口，例如 `package.json` scripts、`Makefile`、`justfile`、CI 命令、`mvnw`、`gradlew` 或其他项目脚本。
- 运行 `git status --short`；若当前目录不是 Git 仓库，或环境不支持 worktree，先明确说明，再在原目录工作。
- 避免把无关改动混入依赖升级批次；能隔离时只暂存当前批次文件，不能安全隔离时先向用户说明风险。

## 识别当前工具链

- 优先使用仓库已经采用的工具，不要只因个人习惯切换包管理器或构建系统。
- 若存在 wrapper，优先使用 wrapper，例如 `./gradlew`、`mvnw` 或仓库定义的统一脚本。
- 若同一仓库存在多个生态或多个子项目，先按生态、模块或工作区拆分，再决定批次，不要把无关生态混进同一次升级。
- 若无法可靠识别统一的依赖检查入口，先给出探测结果和局限，再退回"只规划不执行"或"只处理明确指定依赖"的模式。

## 检查过时依赖

- 依赖检查命令按以下优先级选择：
  1. 仓库已定义的检查脚本、任务或 CI 入口
  2. 当前包管理器或构建工具的原生命令
  3. 仓库已经在使用的生态标准插件或辅助任务
- 常见示例：
  - Node 项目通常使用 `pnpm outdated`、`npm outdated`、`yarn outdated` 或仓库脚本
  - Python 项目通常使用 `poetry show --outdated`、`pip list --outdated`，或仓库对 `uv` / `pip` 的封装命令
  - Java 项目常通过仓库已有的 Maven / Gradle 任务或版本检查插件获取候选升级列表
- 整理结果时，至少记录：依赖名、所在模块、当前版本、声明范围、候选版本、是否属于 major 升级、是否与其他包强耦合。
- 遇到 major 升级、跨多个 major 的升级、明显的兼容性风险，或工具链核心组件升级时，先查官方迁移指南或发行说明，再决定目标版本。

## 规划升级与验证

先分批，再执行。每个批次都要明确写出"升级哪些依赖"和"跑哪些验证"。

按以下顺序优先分批：

1. 开发依赖和辅助工具的低风险 patch/minor 更新
2. 运行时依赖的低风险 patch/minor 更新
3. 同一生态或强耦合工具链的成组更新，例如构建工具、测试框架、lint 生态、框架官方配套插件
4. major 更新；默认一次只处理一个包，或一个强耦合小组

按以下规则拆批：

- 不要把多个无关的 major 更新放进同一批。
- 把存在 `peerDependencies`、插件主从关系、共享 runtime 或共享 lockstep 版本要求的包尽量放到同一批。
- 把框架核心包、构建工具、测试框架、类型系统相关升级视为高风险批次。
- 多模块仓库默认先按模块拆分，再按风险拆批；只有在多个模块必须同步升级时才合并处理。
- 遇到跨生态联动时，默认先拆成更小、更容易回滚和验证的步骤。

按以下规则规划验证：

- 每个批次都运行最小但有效的验证，不要完全跳过验证。
- 涉及构建工具、框架、运行时依赖或类型系统时，至少补充一次构建、编译或等效验证。
- 涉及测试框架、lint 工具或格式化工具时，至少运行对应工具本身和一项受影响的回归验证。
- 最后一批完成后，再运行一轮更完整的回归验证。

## 执行单个批次

对每个批次重复以下流程：

1. 说明该批次准备升级的依赖，以及计划执行的验证。
2. 使用当前生态实际采用的工具或 wrapper 执行升级，不要擅自切换到别的包管理器。
3. 检查依赖清单、锁文件和必要配置文件的变化是否符合预期，例如 `package.json`、`pyproject.toml`、`Cargo.lock`、`go.mod`、`composer.lock`、`pom.xml`、`build.gradle*` 等。
4. 运行该批次预定的验证命令。
5. 如果验证失败，先在当前批次内修复受升级影响的代码、配置或测试。
6. 如果无法在合理范围内修复，撤回当前批次并拆小后重试，不要把失败批次拖进后续提交。
7. 如果验证通过，立即准备提交当前批次；若当前目录不是 Git 仓库，则立即记录清晰的批次结果和后续起点。

## 提交规则

- 每个成功批次必须单独提交，不要等全部升级完成后再集中提交。
- 提交前检查 `git diff --staged`，确认暂存区只包含当前批次相关文件。
- 把为当前批次补充的兼容性修复、配置调整和测试更新与该批次一起提交。
- 优先遵循仓库现有提交规范。
- 仓库没有明确规范时，使用清晰直接的提交信息，例如 `chore: upgrade vite vitest`。
- 如果当前目录不是 Git 仓库，要明确说明无法提交，并用批次摘要代替提交记录。

## 收尾

- 每完成一个批次后，重新运行当前生态的依赖检查命令，确认剩余过时依赖。
- 仍有剩余项时，进入下一批次并重复同样流程。
- 全部完成后，运行最终验证并向用户汇总：
  - 已升级的依赖和对应批次
  - 每个批次执行的验证
  - 每次成功后的提交摘要或批次检查点
  - 刻意保留未升级依赖的原因

## 执行约束

- 默认选择最小可验证批次，而不是最大吞吐批次。
- 默认把 major 更新视为高风险。
- 只在确有必要时修改应用代码、配置或测试；这些修改必须与对应升级批次一起验证、一起提交。
- 如果用户只要求检查或规划，停在计划阶段，不执行升级命令。
- 如果用户要求尽量自动完成，也不要跳过分批、验证和逐批提交。
- 持续向用户同步依赖检查结果、当前批次内容、验证结果和提交进展。
- 如果当前工具无法可靠列出过时依赖，要明确说明局限，并退回可验证的替代方案，不要猜测版本状态。
